## Data Abstraction Layers (A0–A3)

This hierarchy is determined by the data itself.

An describes the **semantic interpretation level** of data, not its storage format. The same data can be interpreted at different levels depending on context — for example, an expanded prose string is physically stored at A0, but its semantics belong to A1. A0 is the most precise; A∞ is the most abstract. Each layer is the input foundation for the layer above it, and the compressed product of the layer below.

| Layer | Meaning in this system | Examples |
|---|---|---|
| **A0** | Exact integers, enums, and strings — fully recoverable | `HP=7`, `depth=3`, `mark: lamp_oil`, prose as a string |
| **A1** | Concrete situation skeleton: scene encounters, option structure — location/atmosphere intent, A1 semantics embedded in prose, no exact values | `scene: market offer`, `option: buy the idol` |
| **A2** | Narrative tendency: state tendency bands and classification labels, finite vocabulary, cannot be precisely recovered | `"sanity: low"`, `"bearing: hollowing"`, `"toll: destabilizing"` |
| **A3** | Creative intent: theme, tone, worldview — C3 uses this to generate constraint structures passed to lower layers | `"dark souls folk horror"`, tone description |
| **A∞** | The system design itself: framework rules, module boundaries, invariants — modified only by C∞ | LANGUAGE.md, architecture.md |

Semantic layer of each type of information in the system:

| Layer | Information |
|---|---|
| **A0** | `HP`, `money`, `sanity`, `depth`, `act`, `max_health`; `active_marks`, `major_events`, `traumas`; `sanity_cost`, effect values; `waypoint_id`, `option_id`, `context_id`, `decision_type`, `entry_id` (stored as strings); expanded prose (stored as strings) |
| **A1** | `scene_concept`, option skeleton (label, intent description); scene skeletons contents; saga waypoint graph (topology, `next_waypoints`); `map_blurb`; `narration_table.json` (per-saga, generated by C3) |
| **A2** | Bearing entries (arc-state catalog); `toll` labels; toll lexicon entries; tendency descriptions (`"sanity: low"`); `rules.json` rules (thematic constraints) |
| **A3** | Saga theme string; tone description; worldview / narrative register |
| **A∞** | LANGUAGE.md, architecture.md; the An/Cn framework itself |

**These terms exist at different levels of semantic interpretation. This table assigns each term an appropriate type for the system to process.**

The higher the layer, the lower the rate of change, the longer the lifecycle, and the more resistant it is to disruption by lower-layer events: A0 changes every encounter, A1 is fixed per-waypoint, A2 is fixed per-saga, A3 is written by C∞ (humans) and rarely changes thereafter. This is the foundation for higher layers constraining lower ones — frequent low-layer changes do not propagate upward to redefine high-layer meanings.

## Processing Cores (C0–C3)

This hierarchy is determined by the core itself.  
A Cn core efficiently understands and processes information up to level An, but higher-order Cn cores are substantially more expensive.

Cn naming reflects **the highest abstraction level required to drive decisions**. Two core rules:

- **Cn may reference any information at An and below. Decisions that change Am-layer data must come from C(m+1) or higher; execution may be delegated to a lower Cn.** For example, C2 reads A0 values as reference, decides effects, then delegates writing to C0. A2-layer data is decided by C3; C2 is read-only at that layer.
- **Cn does not need to understand information above An.** Higher-core decisions are black-box instructions to lower cores — C0 executes `sanity -= 2` without knowing the narrative tendency behind it; C1 expands a scene skeleton without knowing what a bearing is.

In this system: C0 = deterministic processor, C1 = qwen2.5:7b (local), C2 = Haiku, C3 = Opus.

| Name | Model | Reads | Outputs | When |
|---|---|---|---|---|
| **C0** | Local deterministic processor (no LLM) | A0 + C2 instructions | A0 (executes effects, updates precise state) | Runtime |
| **C1** | qwen2.5:7b (local ollama) | A1 skeleton + C2 instructions | A0 (prose strings) | Runtime |
| **C2** | claude-haiku-4-5 | A2 (tendency framework) | A0 (effect values) + A1 (tolls, bearing) | Offline (`gen`) + Runtime |
| **C3** | claude-opus-4-6 | A3 (creative intent) | A2 (cache table, toll lexicon, rules) + A1 (saga graph, waypoint structure, narration_table) | Offline (`gen`) |
| **C∞ (human)** | — | A∞ (any layer) | A∞ (any layer) | Offline, drives C3 |

The system's goal is to **minimize C∞ intervention frequency** by delegating work to the lowest sufficient Cn. C∞ intervenes once (writes A3 intent); C3 translates that into lower-layer constraints; C2/C1/C0 then operate autonomously. **This matters because Cn is substantially expensive when n is large.**

## Analogy: Nervous System Hierarchy

Loombound's three-tier processing architecture maps almost exactly onto the hierarchical structure of the biological nervous system:

| Biological tier | Function | Loombound equivalent | Latency |
|---|---|---|---|
| Cerebral cortex (central) | Slow, deep reasoning, long-term planning | C3 — Claude Opus (offline saga generation) | Seconds to minutes |
| Spinal reflex arc | Fast, local, simple signal processing | C2 — Claude Haiku (runtime bearing classification) | 1–2 seconds |
| Peripheral nerves + muscle | Pure local execution, no network | C1 — qwen2.5:7b local (scene expansion) | 2–10 seconds (ideal) |

**Why the analogy runs deep:**

- The brain (C3/Opus) builds the world model and generates complex structures — activated only "offline," never in the runtime loop
- The spinal cord (C2/Haiku) handles fast runtime classification and signal routing — no brain involvement needed, completes in 1–2 seconds
- The muscle (C1 local) produces the final output — fully local, no network, latency determined by hardware not API

**Why Loombound's C1 is slow in practice:** The current 16–37 second latency reflects the author running inference on underpowered local hardware with no GPU, not the model itself. qwen2.5:7b requires ~4–5 GB VRAM for full-precision loading. With a GPU it reaches 20–80 token/s (~2–10 seconds per scene expansion); without one it falls back to CPU inference, which is 5–10× slower. With adequate hardware, C1 latency falls into the same 1–2 second range as C2 — and the "muscle reflex" analogy holds fully.

The nervous system separates slow reasoning from fast reflexes as a result of evolutionary optimization. Loombound's layered architecture is the same engineering optimization: route each task to the cheapest sufficient processing tier, and never invoke the most expensive tier when a simpler one will do.

## Cache Layers (Four Distinct Things)

Cache table naming corresponds to the abstraction level of the data stored: arc-state catalog holds A2-layer data, used by C2; scene skeletons holds A1-layer data, used by C1 (C2 uses the subset A1 option index).

| Name | Storage | Generated | Lifecycle | Purpose |
|---|---|---|---|---|
| **arc-state catalog** | `data/arc_state_catalog.json` | One-time (C3) | Global, across sagas | Bearing enumeration (~50 entries), C2 runtime prompt cache |
| **scene skeletons** | `data/waypoints/<id>/scene_skeletons.json` | Each `gen` (C2) | Per-saga | Per-waypoint scene skeletons: scene_concept, option structure, effects |
| **A1 option index** | Not stored — derived at runtime | Runtime startup | Per-session | scene skeletons with effects stripped, used as C2 per-saga cached prefix |
| **PrefetchCache** | In-memory, not persisted | Runtime | Per-session | Prefetched waypoint content + C2-returned per-option effects and tolls |

> **Anthropic prompt cache** is an API-level token cache. It is unrelated to the four concepts above and does not appear in this table.

> **arc-state catalog vs scene skeletons**: The two are orthogonal, not a subset relationship. arc-state catalog is the "bearing vocabulary" (global — C2 uses it to classify the current narrative trajectory). scene skeletons is "the scene skeleton for this saga" (per-saga — C1 uses it to expand text; C2 uses A1 option index to know the current options).

## Domain Terms

| Term | Definition |
|---|---|
| **saga** | One generated narrative graph: waypoint topology, tone, initial state, toll lexicon |
| **waypoint** | A location in the graph, with a node_type, depth, and N encounters |
| **encounter** | One decision moment: context + options → player choice → result (including effects) |
| **toll** | The consequence label for a single option (`stable` / `destabilizing` / theme word), generated by C2 at runtime |
| **toll lexicon** | Saga-specific toll vocabulary, generated by C3, stored as `data/sagas/<id>_toll_lexicon.json`, cached after A1 option index at runtime |
| **bearing** | The classification result of the current narrative trajectory, expressed as an `entry_id` in the arc-state catalog |
| **tendency** | A0 precise values compressed into narrative tendency (low/moderate/high) and sent to C2 as text descriptions (A2-layer data) |
| **depth** | The depth level of a waypoint in the saga graph — strictly increasing along any path |
| **mark** | A persistent status label attached to the player (e.g. `lamp_oil`, `warding_tools`) |

## Workflow Phases

| Phase | Command | Core | Output |
|---|---|---|---|
| **arc palette** | `./loombound arc-palette` | C3 (one-time) | `data/arc_state_catalog.json` |
| **gen** | `./loombound gen "theme"` | C3 + C2 | `data/sagas/<id>.json` + `data/sagas/<id>_toll_lexicon.json` + `data/sagas/<id>_rules.json` + `data/sagas/<id>_narration_table.json` + `data/waypoints/<id>/scene_skeletons.json` |
| **run** | `./loombound run` | C2 + C1 | Game session — no persisted output |
| **report** | `./loombound report` | None | Token usage report parsed from `logs/llm.md` |
