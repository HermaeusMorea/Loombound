# Decision Log

## 文档定位

本文档用于记录关键架构与产品决策，说明“为什么项目现在这样定”。它不是 roadmap，也不是任务列表；它只记录已经接受的取舍、背景与后果，用于防止未来自己或 AI 在迭代中把项目带偏。

## 使用原则

- 只记录已经接受的决定
- 每条记录都应说明背景、决策内容与 tradeoff
- 若未来改口，应新增后续决策，不覆盖历史
- 阶段计划请写入 roadmap、backlog 或其他执行文档

## 决策 001

### 标题

先做 Python 离线原型，不直接做正式 mod

### 状态

`accepted`

### 日期

2026-04-12

### 背景

项目当前最不确定的部分，不是如何接入游戏，而是“礼官导演系统”本身是否有趣、是否能稳定运作、是否有足够清晰的数据结构。若一开始就直接做正式 mod，会过早被 mod API、游戏 UI、运行时限制和工程耦合绑住。

### 决策内容

当前仓库定位为 Python 原型仓库。优先完成离线 sample contexts、deterministic rule engine、CLI/demo、文档与测试；真实游戏接入放到后续阶段，并且先从只读验证开始。

### 后果 / Tradeoff

- 好处：可以先验证玩法与结构，而不是先背工程债
- 好处：方便迭代 schema、规则模板与旁白格式
- 代价：短期内缺乏真实游戏内手感验证
- 代价：需要额外设计一层 adapter，未来再映射到真实游戏

## 决策 002

### 标题

deterministic-first，LLM optional

### 状态

`accepted`

### 日期

2026-04-12

### 背景

本项目中的规则、约束、惩罚与日志，必须可执行、可复现、可调试。如果让 LLM 直接决定规则生效、惩罚数值或合法动作范围，系统会失去稳定性，也无法为未来 mod 化提供可靠基础。

### 决策内容

当前核心逻辑全部采用决定论结构：状态标准化、signal 提取、主题打分、规则匹配、规则选择、惩罚结算、日志记录都由代码与模板负责。LLM 仅作为未来可选层，用于旁白润色、总结与解释性 rerank，不参与规则是否生效的最终裁定。

### 后果 / Tradeoff

- 好处：输出稳定，容易做回归测试与 snapshot 审查
- 好处：更适合未来与真实游戏状态对接
- 好处：便于作者调权重、调规则与查错
- 代价：前期文案表现力有限
- 代价：需要人工维护更多模板与标签

## 决策 003

### 标题

第一版只做战斗外决策

### 状态

`accepted`

### 日期

2026-04-12

### 背景

战斗内限制牵涉出牌顺序、敌我状态、时点规则、复杂组合与即时反馈，建模复杂度远高于战斗外节点。相比之下，地图、卡牌奖励、商店、事件等选择点更适合先验证“礼官法令”是否有趣。

### 决策内容

MVP 只覆盖战斗外决策：地图选路、卡牌奖励、商店、事件，并预留 `relic_choice` 作为可兼容场景。战斗内系统属于未来议题，不进入当前原型主线。

### 后果 / Tradeoff

- 好处：更容易定义清晰输入与可验证输出
- 好处：更快形成第一批可试玩规则
- 好处：更容易让礼官风格先成立
- 代价：暂时无法回答“战斗内礼制约束是否更有戏剧性”
- 代价：最终 mod 化时仍需另开一条更复杂的设计线

## 决策 004

### 标题

先软惩罚，不先硬锁

### 状态

`accepted`

### 日期

2026-04-12

### 背景

如果一开始就直接禁点、强制路径或强制拿牌，系统会显得过于武断，也不利于观察玩家真实反应。原型阶段更重要的是知道：玩家是否愿意为了收益而违礼，系统的“节目效果”是否成立，礼崩曲线是否合理。

### 决策内容

第一版允许玩家违礼。违礼行为会被标记、记录，并增加 `ritual_collapse`。后续规则可逐步收紧，但当前不以硬锁为优先，也不直接替玩家点击。

### 后果 / Tradeoff

- 好处：玩家保有主动性，反馈更真实
- 好处：更方便调试规则与惩罚曲线
- 好处：节目效果更强，系统人格更鲜明
- 代价：短期内“规则威慑力”较弱
- 代价：需要后续设计礼崩如何真正转化为后果

## 决策 005

### 标题

规则模板优先于自由生成规则

### 状态

`accepted`

### 日期

2026-04-12

### 背景

本项目的规则不是装饰文本，而是需要真正落到可执行约束上。如果规则由模型自由生成，将难以保证行为边界、冷却、冲突消解、惩罚力度与未来 UI 呈现的一致性。

### 决策内容

规则以结构化模板为主，内容作者通过模板字段表达触发条件、限制对象、惩罚、文案槽位、冷却与黑名单条件。模型可以参与未来的内容整理或文本润色，但不直接发明生效规则，也不改写既有 rule id 与 penalty 语义。

### 后果 / Tradeoff

- 好处：规则稳定、可测试、便于调惩罚
- 好处：更适合长期扩内容与未来接 mod UI
- 好处：便于统计覆盖率与排查冲突
- 代价：前期内容编写成本更高
- 代价：需要更认真地设计标签体系与模板字段

## 决策 006

### 标题

系统从即时裁决器升级为每局记忆型礼官系统

### 状态

`accepted`

### 日期

2026-04-12

### 背景

如果系统只在当前节点上即时裁决，而不记录玩家此前的守礼、违礼、贪利、冒险或反覆行为，那么礼官更像一次性脚本，而不像持续存在的制度人格。单次裁决足以验证局部玩法，但不足以形成“案卷感”“累犯感”与连续的礼官方向。

### 决策内容

项目的长期系统定位升级为 run-scoped ritual judge with memory。默认由 kernel 维护结构化 `run memory`，记录礼崩值、最近法令、最近违礼记录、主题计数、誓约状态与礼官态度参数，并让这些记忆反馈进入后续裁决。未来可在此基础上叠加 LLM judge memory / persona memory，但后者不取代结构化记忆作为核心事实层。

### 后果 / Tradeoff

- 好处：规则更像制度而不是随机旁白
- 好处：惩罚与语气可以形成连续性
- 好处：更自然地支持未来的 persona memory 与 LLM 导演能力
- 代价：需要定义 memory schema、更新时机与反馈机制
- 代价：测试不再只看单点输入输出，还要验证跨节点状态演化

## 决策 007

### 标题

采用 overlay architecture，而不替代原生状态机

### 状态

`accepted`

### 日期

2026-04-12

### 背景

原生游戏已经拥有原生 run-level 与 encounter-level state machines。若本项目未来接入真实游戏，仍不应假定自己替代这些原生状态机。

### 决策内容

项目采用 overlay architecture。未来接入阶段应通过 adapters / hook points 观察原生流程，将其整理为 `EventTrace`，并在此基础上更新 `MetaState`、生成有界的 `ProposedEffects`；本项目不替代原生状态机。

### 后果 / Tradeoff

- 好处：边界更清楚，更适合 replay / debug
- 好处：更容易保持原生流程与 overlay 系统解耦
- 代价：需要更仔细地设计事件 schema 与 integration boundary

## 决策 008

### 标题

默认模式下，LLM 只更新 `MetaState` 并提出 `ProposedEffects`

### 状态

`accepted`

### 日期

2026-04-12

### 背景

若允许 LLM 直接变更核心游戏状态，会削弱决定论、可调试性与可回放性。

### 决策内容

默认模式下，LLM 可以解释 `EventTrace`、更新 `MetaState`、生成 `ProposedEffects`，但最终涉及 `CoreState` 的 mutation 必须经过决定论验证与 `EffectApplier`。

### 后果 / Tradeoff

- 好处：行为更少 freeform，但架构更安全
- 好处：更利于未来 mod integration
- 代价：LLM 的直接控制感较弱
