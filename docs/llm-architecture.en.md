# Loombound LLM Architecture

## Design Principles

The LLM is a required content layer but does not replace the deterministic kernel. Responsibility boundaries:

| Owner | Responsibility |
|---|---|
| **kernel** | State structure, valid updates, rule selection, replay consistency |
| **LLM** | Content asset generation, scene text expansion, arc classification, memory summarization |

All LLM output must be normalized through `state_adapter` before entering the runtime.

---

## IRIS Three-Layer Architecture

This design references the PRISM protocol, splitting the AI layer into three tiers that each speak a different language:

```
┌────────────────────────────────────────┐
│  M2 / Slow Core — Claude Opus          │  quasi language: arc tendency, narrative direction
│  Runtime arc classifier (offline plan) │
└──────────────────┬─────────────────────┘
                   │  Table A + B (preloaded offline)
┌──────────────────▼─────────────────────┐
│  M1 / Fast Core — gemma3:4b (local)    │  quasi ↔ precise text conversion
│  Scene text expansion (runtime)        │
└──────────────────┬─────────────────────┘
                   │  structured payloads
┌──────────────────▼─────────────────────┐
│  M0 / Kernel — deterministic core      │  precise language: integers, enums, valid schema
│  RunMemory / NodeMemory                │
└────────────────────────────────────────┘
```

Forcing Slow Core and Kernel to communicate directly causes two failure modes:
- Kernel forced to accept ambiguous input → validation breaks
- Slow Core forced to process exact numeric values → semantic distortion

---

## M0 — Kernel Layer (deterministic)

Corresponds to `RunMemory` / `NodeMemory`, fully deterministic updates.
No LLM involvement; the factual foundation of the system.

---

## M2 — Slow Core (Claude Opus, runtime arc classifier)

### Offline: generating Table A and Table B

**Table A (global arc-state palette, one-time)**

50 enumerated entries across four dimensions:
- `arc_trajectory`: rising / plateau / climax / resolution / pivot
- `world_pressure`: low / moderate / high / critical
- `narrative_pacing`: slow / steady / accelerating / sprint
- `pending_intent`: exploration / confrontation / revelation / recovery / transition

Stored at `data/m2_table_a.json`, loaded into prompt cache at runtime.

**Table B (per-campaign scene skeletons, auto-generated on each `gen`)**

One scene skeleton per node, generated by Claude Haiku in batches of 3:
- `scene_concept`: core scene direction (20–40 words)
- `sanity_axis`: sanity pressure axis for this node
- `options`: option intents, tags, effects

Stored at `data/nodes/<campaign_id>/table_b.json`.

### Runtime: arc classification (preloaded path)

When entering each node, a background thread calls the M2 Classifier:

```
Table A (cached) + M1/M0 quasi state → entry_id (integer)
```

Output is very short (~10 tokens); after cache hit the cost is minimal.

---

## Prompt Cache Strategy

The M2 Classifier system prompt expands into a full classification guide (~4,644 tokens),
exceeding claude-opus-4-6's cache minimum threshold (4,096 tokens).

```
call1: cache_created=4644  input=104   ← cache written
call2: cache_created=0     cache_read=4644  input=104   ← cache hit
```

After a hit, each call bills only ~104 tokens (the quasi state portion), saving ~90%.

---

## M1 — Fast Core (gemma3:4b, local)

Receives the M2 classification result + Table B scene skeleton and expands them into the full text the player sees:

| Field | Source | Fast Core expansion |
|---|---|---|
| `scene_summary` | scene_concept | 3–5 sentences of atmospheric prose |
| `sanity_question` | sanity_axis | 1 suspense question |
| option labels | intent | 5–10 word option text |
| `add_events` | intent + effects | 1–2 sentences of causal record |

Runs locally, zero API cost.

---

## Preloaded Path vs Dynamic Path

### Preloaded path (with Table A + B, recommended)

```
While playing Node N, background thread:
  1. build_quasi_description()  ← M1/M0 state summary
  2. M2 Classifier → entry_id
  3. M2Store.lookup_seed(entry_id) → Table B skeleton
  4. Fast Core.expand(skeleton, state) → payloads
  5. Store in PrefetchCache

Entering Node N+1:
  PrefetchCache.consume() → use pre-generated content directly
```

### Dynamic path (no Table B, fallback)

```
  SlowCoreClient.plan_node()  ← runtime LLM call to generate content
  → Fast Core expansion
```

## Quasi Language

State passed from M0 → M2 uses "quasi-precise descriptions", example:

```
## Current state
  health:  high (stable)
  sanity:  moderate (depleting)
  floor:   2,  act: 2

## Scene history (M1 — last 3 nodes)
  [1] dark_alley — turbulent, pressure=high, trajectory=accelerating
  [2] archive    — stable, pressure=moderate, trajectory=rising

Classify the arc state that best matches the current game state.
```

M2 returns `{"entry_id": N}` or `{"entry_id": -1}` (falls back to authored content on no match).

## Content Boundaries

### LLM may generate

- Campaign graph, node topology, map_blurb (Opus, `./loombound gen`)
- scene_concept, sanity_axis, options (Haiku, Table B)
- scene_summary, sanity_question, option labels (gemma3, runtime expansion)
- Arc classification entry_id (Opus M2 Classifier)

### LLM may not write directly

- CoreState numeric fields (health / money / sanity delta computed by kernel)
- Final RuleTemplate selection
- Structural fields of Run / Node / Arbitration

## Offline Generation Flow

```bash
# 1. Generate global arc palette (one-time)
./loombound arc-palette

# 2. Generate campaign (Opus builds graph + Haiku builds Table B, automatic)
./loombound gen "theme" --lang zh

# 3. Runtime uses preloaded path
./loombound run --slow anthropic --lang zh
```

### Cost Analysis

#### Real data: campaign "Heritage Liquidation Bureau" (5 nodes, 11 arbitrations)

> Full LLM log and `./loombound report` output: [logs/heritage_liquidation_bureau_playthrough.md](../logs/heritage_liquidation_bureau_playthrough.md)

**Offline phase (one-time, amortized across all players)**

| Step | Model | Actual cost |
|---|---|---|
| Campaign graph generation | Claude Opus | $0.0454 |
| Table B scene skeleton generation | Claude Haiku | $0.0166 |
| **Offline subtotal** | | **$0.0620** |

**Runtime (per play)**

| Step | Model | Actual cost | Details |
|---|---|---|---|
| M2 classification × 3 nodes (parallel, first-time cache_created) | Claude Opus | $0.0045 | input ≈ 106–168 tokens, cache_created=4145 |
| M2 classification × 2 nodes (later preload, cache_read hit) | Claude Opus | $0.0090 | input ≈ 304 tokens, cache_read=4145, $0.0045/call |
| Fast Core expansion × 11 arbitrations | gemma3 (local) | **$0** | prompt=13,210 + eval=6,783 = 19,993 tokens |
| **Runtime subtotal** | | **$0.0135** | |

**First play total: $0.0754**

M2 prompt cache savings: ~$0.0373 (8,290 cache_read tokens — Table A stays warm after the initial parallel calls build it)

---

#### Cost per play approaches the runtime floor as player count grows

The offline cost is one-time and amortizes across all players:

| Plays | Total API spend | Per-play average |
|---|---|---|
| 1 | $0.0754 | $0.0754 |
| 10 | $0.197 | $0.0197 |
| 100 | $1.42 | $0.0142 |
| ∞ | — | **$0.0135** |

**Runtime floor: $0.0135 per play** (M2 classification only — Fast Core is always free)

---

#### What if Fast Core were replaced with Opus?

These 11 arbitration expansions averaged 617 eval tokens each, with roughly 300 tokens of input. The Opus-equivalent cost:

```
Per arbitration: 300 input + 617 output
= 300 × $5/M + 617 × $25/M ≈ $0.017
11 arbitrations ≈ $0.187/play
+ M2: $0.0135
Opus-only runtime per play: ≈ $0.20
```

| Plays | Current architecture | Opus-only | Ratio |
|---|---|---|---|
| 1 | $0.075 | $0.24 | 3.2× |
| 10 | $0.20 | $2.5 | 12.5× |
| 100 | $1.42 | $20.5 | 14.4× |
| 1000 | $13.6 | $200 | **14.7×** |

As player count scales, the cost advantage stabilizes at **~15×** — a direct consequence of the runtime cost ratio ($0.0135 vs $0.20).

> **Note:** The calculations above use the pricing constants in the codebase ($5/$25, matching Claude 3 Opus). The actual `claude-opus-4-6` price is $15/$75; at real prices the gap is approximately **44×**.
