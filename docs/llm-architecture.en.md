# Loombound LLM Architecture

## Design Principles

The LLM is a required content layer but does not replace the deterministic kernel. Responsibility boundaries:

| Owner | Responsibility |
|---|---|
| **kernel** | State structure, valid updates, rule selection, replay consistency |
| **LLM** | Content asset generation, scene text expansion, arc classification, option value assignment |

All LLM output must be normalized through `state_adapter` before entering the runtime.

---

## IRIS Three-Layer Architecture

This design references the PRISM protocol, splitting the AI layer into three tiers that each speak a different language:

```
┌────────────────────────────────────────┐
│  M2 — Claude Haiku                     │  quasi language: arc tendency, per-option values
│  Runtime arc classifier (per-choice)   │
└──────────────────┬─────────────────────┘
                   │  entry_id + effects (injected dynamically)
┌──────────────────▼─────────────────────┐
│  M1 / Fast Core — gemma3:4b (local)    │  quasi ↔ precise text conversion
│  Scene text expansion (runtime)        │
└──────────────────┬─────────────────────┘
                   │  structured payloads
┌──────────────────▼─────────────────────┐
│  M0 / Kernel — deterministic core      │  precise language: integers, enums, valid schema
│  RunMemory / NodeMemory                │
└────────────────────────────────────────┘
```

---

## M0 — Kernel Layer (deterministic)

Corresponds to `RunMemory` / `NodeMemory`, fully deterministic updates.
No LLM involvement; the factual foundation of the system.

---

## M2 — Runtime Arc Classifier (Claude Haiku)

### Offline: generating Table A and Table B

**Table A (global arc-state palette, one-time, generated by Opus)**

~50 enumerated entries across four dimensions:
- `arc_trajectory`: rising / plateau / climax / resolution / pivot
- `world_pressure`: low / moderate / high / critical
- `narrative_pacing`: slow / steady / accelerating / sprint
- `pending_intent`: exploration / confrontation / revelation / recovery / transition

Stored at `data/m2_table_a.json`, loaded into Haiku's prompt cache at runtime.

**Table B (per-campaign scene skeletons, generated by Haiku on each `gen`)**

One scene skeleton per node:
- `scene_concept`: core scene direction (20–40 words)
- `sanity_axis`: sanity pressure axis for this node
- `options`: option intents (intent), tags — **no h/m/s values**

Stored at `data/nodes/<campaign_id>/table_b.json`.

**Table C (derived at runtime, not stored as a file)**

Extracted from Table B: `option_id + intent` only, no numeric values. Used as Haiku's per-campaign cached prefix. Haiku looks up this structure and fills in h/m/s values for each option.

### Runtime: per-choice classification + value assignment

After each player choice, a background thread immediately calls the M2 Classifier:

```
Table A (cached) + Table C (cached) + M1/M0 quasi state + next arbitration identifier
  → entry_id (integer) + effects ({option_id: {h, m, s}})
```

- **entry_id**: updates the local `_current_arc_id`, used by gemma3 prefetch to look up Table A tendency
- **effects**: injected into option values before the next arbitration is displayed
- gemma3 prefetch reads `_current_arc_id` directly — no active Haiku call needed

Last choice in a node: only entry_id returned (no effects).  
After the player selects the next node, the main loop triggers a Haiku call for that node's first arbitration.

## Prompt Cache Strategy

The Haiku M2 Classifier cache has three layers:

```
system prompt (classification guide + effect assignment rules)  ~3,000 tokens  global cache
tool schema (select_arc_and_effects)                            ~1,000 tokens  global cache
Table A                                                         ~1,500 tokens  session cache
Table C (per-campaign option structure)                         ~2,000 tokens  per-campaign cache
---
dynamic portion (quasi state + target arbitration hint)           ~300 tokens  billed each call
output (entry_id + one arb's effects)                              ~80 tokens  billed each call
```

Per-call cost after cache hit:

| Portion | Tokens | Rate | Cost |
|---|---|---|---|
| Cache read (~7,500 tokens) | 7,500 | $0.08/M | ~$0.0006 |
| Dynamic input | 300 | $0.80/M | ~$0.00024 |
| Output | 80 | $4/M | ~$0.00032 |
| **Total** | | | **~$0.001** |

## M1 — Fast Core (gemma3:4b, local)

Receives the arc tendency (from Table A entry_id) + Table B scene skeleton and expands them into the full text the player sees:

| Field | Source | Fast Core expansion |
|---|---|---|
| `scene_summary` | scene_concept | 3–5 sentences of atmospheric prose |
| `sanity_question` | sanity_axis | 1 suspense question |
| option labels | intent | 5–10 word option text |
| `add_events` | intent | 1–2 sentences of causal record |

h/m/s values are injected by Haiku; gemma3 uses effect direction as context for add_events but does not output raw numbers.

Runs locally, zero API cost.

## Runtime Data Flow

```
Game start
  └─ _current_arc_id = 0 (fixed initial value)
  └─ gemma3 warmup (background)

Enter node N
  └─ prefetch.trigger(N+1)
       └─ read _current_arc_id → look up Table A tendency
       └─ look up Table B skeleton
       └─ gemma3 expansion (background, parallel with player gameplay)
  └─ after player picks node, main loop fires Haiku(N+1, arb=0) (background)

Play Arb 0
  └─ consume_arb_effects(N, 0) → wait for Haiku result → inject effects
  └─ display options → player chooses
  └─ update_arc_state(quasi, N, next_arb=1) (background)

Play Arb 1
  └─ consume_arb_effects(N, 1) → wait for Haiku result → inject effects
  └─ player chooses (last arb)
  └─ update_arc_state(quasi, None, None) → update entry_id only

Node N ends, player selects node N+1
  └─ update_arc_state(quasi, N+1, arb=0) (background)
  └─ enter node N+1, prefetch result already ready
```

## Quasi Language

State passed from M0 → M2 uses "quasi-precise descriptions", example:

```
## Current state (quasi)
  health:  high (stable)
  sanity:  moderate (falling)
  floor:   2,  act: 2
  dominant themes: dread×3, isolation×2

## Node trajectory (2 completed)
  [crossroads] floor=1  sanity_delta=-1  flags=none
  [market]     floor=2  sanity_delta=-2  flags=witnessed_violence

## Active node so far (partial)
  arbitrations_resolved=1  sanity_lost=1

Assign effects for: node_id=archive_vault, arb_index=1
```

Haiku returns `{"entry_id": N, "effects": [{"id": "opt_a", "h": -2, "m": 0, "s": -1}, ...]}`.

## Content Boundaries

### LLM may generate

- Campaign graph, node topology, map_blurb (Opus, `./loombound gen`)
- scene_concept, sanity_axis, option intents (Haiku, Table B generation)
- Arc state entry_id + per-option h/m/s values (Haiku, runtime M2)
- scene_summary, sanity_question, option labels, add_events (gemma3, runtime expansion)

### LLM may not write directly

- CoreState numeric field application (executed by kernel)
- Final RuleTemplate selection
- Structural fields of Run / Node / Arbitration

## Offline Generation Flow

```bash
# 1. Generate global arc palette (one-time)
./loombound arc-palette

# 2. Generate campaign (Opus builds graph + Haiku builds Table B, automatic)
./loombound gen "theme" --lang zh

# 3. Play
./loombound run --lang zh
```

## Cost Analysis

### Reference data: campaign "Heritage Liquidation Bureau" (5 nodes, 11 arbitrations)

> Note: the runtime figures below were collected before the Haiku M2 switch (the old Opus per-node design). Offline costs remain the same.

**Offline phase (one-time, amortized across all players)**

| Step | Model | Actual cost |
|---|---|---|
| Campaign graph generation | Claude Opus | $0.0454 |
| Table B scene skeleton generation | Claude Haiku | $0.0166 |
| **Offline subtotal** | | **$0.0620** |

**Runtime estimate (Haiku per-choice design)**

| Step | Model | Estimated cost |
|---|---|---|
| M2 classification × 11 choices (cache hit) | Claude Haiku | ~$0.011 |
| Fast Core expansion × 11 arbitrations | gemma3 (local) | **$0** |
| **Runtime subtotal** | | **~$0.011** |

**Estimated first-play total: ~$0.073**

### Cost per play approaches the runtime floor as player count grows

| Plays | Total API spend | Per-play average |
|---|---|---|
| 1 | ~$0.073 | ~$0.073 |
| 10 | ~$0.173 | ~$0.017 |
| 100 | ~$1.17 | ~$0.012 |
| ∞ | — | **~$0.011** |

---

### What if Fast Core were replaced with Opus?

These 11 arbitration expansions averaged ~617 eval tokens each with ~300 tokens of input. The Opus-equivalent cost at real `claude-opus-4-6` pricing ($15/$75):

```
Per arbitration: 300 × $15/M + 617 × $75/M ≈ $0.051
11 arbitrations ≈ $0.56/play
```

| Plays | Current architecture | Opus-only | Ratio |
|---|---|---|---|
| 1 | ~$0.073 | ~$0.62 | ~8.5× |
| 100 | ~$1.17 | ~$56 | **~48×** |

## Module Locations

```
src/core/llm_interface/
├── m2_classifier.py    ← runtime arc classifier (Haiku, per-choice)
├── prefetch.py         ← background preload threads + arc state tracking
├── fast_core.py        ← gemma3 scene text expansion
├── collector.py        ← M0 → quasi state construction
└── types.py            ← SeedPack / PrefetchEntry data types

src/core/memory/
├── m2_store.py         ← Table A / Table B / Table C loading and lookup
└── types.py            ← M1Entry, M1Store, M2Entry, M2Store

scripts/
├── generate_arc_palette.py   ← generate Table A (Opus)
└── generate_campaign.py      ← generate campaign graph (Opus) + Table B (Haiku)
```
