# Loombound LLM Architecture

## Design Principles

The LLM is a required content layer but does not replace the deterministic kernel. Responsibility boundaries:

| Owner | Responsibility |
|---|---|
| **kernel (C0)** | State structure, valid updates, rule selection, replay consistency |
| **LLM (C1/C2/C3)** | Content asset generation, scene text expansion, bearing classification, option value assignment |

All LLM output must be normalized through `state_adapter` before entering the runtime.

---

## Three-Layer Processing Architecture (C0–C2)

```
┌────────────────────────────────────────┐
│  C2 — Claude Haiku                     │  tendency → bearing + per-option toll/effects
│  Runtime bearing classifier (per-choice)│
└──────────────────┬─────────────────────┘
                   │  entry_id + effects + tolls (dynamically injected)
┌──────────────────▼─────────────────────┐
│  C1 — qwen2.5:7b (local)                │  A1 skeleton → precise prose
│  Scene text expansion (runtime)        │
└──────────────────┬─────────────────────┘
                   │  structured payloads
┌──────────────────▼─────────────────────┐
│  C0 — deterministic core               │  precise language: integers, enums, valid schema
│  RunMemory / WaypointMemory            │
└────────────────────────────────────────┘
```

---

## C0 — Kernel Layer (deterministic)

Corresponds to `RunMemory` / `WaypointMemory`, fully deterministic updates.  
No LLM involvement; the factual foundation of the system.

---

## C2 — Runtime Bearing Classifier (Claude Haiku)

### Offline: generating A2 cache table and A1 cache table

**A2 cache table (global bearing enumeration, one-time, generated by C3/Opus)**

~50 enumerated entries across four dimensions:
- `arc_trajectory`: rising / plateau / climax / resolution / pivot
- `world_pressure`: low / moderate / high / critical
- `narrative_pacing`: slow / steady / accelerating / sprint
- `pending_intent`: exploration / confrontation / revelation / recovery / transition

Stored at `data/a2_cache_table.json`, loaded into Haiku's prompt cache at runtime.

**A1 cache table (per-saga scene skeletons, generated by C2/Haiku on each `gen`)**

One scene skeleton per waypoint:
- `scene_concept`: core scene direction (20–40 words)
- `sanity_axis`: sanity pressure axis for this waypoint
- `options`: option intents, tags — **no h/m/s values**

Stored at `data/waypoints/<saga_id>/a1_cache_table.json`.

**A1 option index (derived at runtime, not stored)**

Extracted from the A1 cache table: `option_id + intent` only, no numeric values.  
Used as Haiku's per-saga cached prefix. Haiku fills in h/m/s values and tolls for each option.

### Runtime: per-choice bearing classification + value assignment

After each player choice, a background thread immediately calls the C2 classifier:

```
A2 cache table (cached) + A1 option index (cached) + toll lexicon (cached)
  + tendency (compressed A0) + next encounter identifier
  → entry_id (integer) + effects ({option_id: {h, m, s}}) + tolls
```

- **entry_id**: updates the current bearing, used by C1 prefetch to look up A2 cache table tendency
- **effects**: injected into option values before the next encounter is displayed
- **tolls**: per-option toll labels, passed to rule enforcement
- C1 prefetch reads the current bearing directly — no active C2 call needed

Last choice in a waypoint: only entry_id returned (no effects).  
After the player selects the next waypoint, the main loop triggers a C2 call for that waypoint's first encounter.

## Prompt Cache Strategy

The C2 classifier cache has four layers:

```
system prompt (classification guide + toll assignment rules)   ~3,000 tokens  global cache
tool schema (arc update + effects)                             ~1,000 tokens  global cache
A2 cache table                                                 ~1,500 tokens  session cache
A1 option index + toll lexicon (per-saga)                      ~2,500 tokens  per-saga cache
──────────────────────────────────────────────────────────────────────────────────────────
tendency (compressed A0) + target encounter hint                 ~300 tokens  billed each call
output (entry_id + one set of effects + tolls)                   ~100 tokens  billed each call
```

Per-call cost after cache hit (measured average over 8 calls):

| Portion | Tokens | Rate | Cost |
|---|---|---|---|
| Cache read (~8,087 tokens) | 8,087 | $0.08/M | ~$0.00065 |
| Dynamic input | ~398 | $0.80/M | ~$0.00032 |
| Output | ~220 | $4/M | ~$0.00088 |
| **Total** | | | **~$0.0019** |

## C1 — Scene Expander (qwen2.5:7b, local)

Receives the current bearing (from A2 cache table entry_id) + A1 cache table scene skeleton and expands them into the full text the player sees:

| Field | Source | C1 expansion |
|---|---|---|
| `scene_summary` | scene_concept | 3–5 sentences of atmospheric prose |
| `sanity_question` | sanity_axis | 1 suspense question |
| option labels | intent | 5–10 word option text |
| `add_events` | intent | 1–2 sentences of causal record |

h/m/s values and tolls are injected by C2; C1 uses effect direction as context for add_events but does not output raw numbers.

Runs locally, zero API cost. `num_predict = -1` (unlimited, prevents truncation on complex scenes).

## Runtime Data Flow

```
Game start
  └─ bearing = 0 (fixed initial value)
  └─ C1 (qwen2.5:7b) warmup (background)

Enter waypoint N
  └─ prefetch.trigger(N)
       └─ read bearing → look up A2 cache table tendency
       └─ look up A1 cache table skeleton
       └─ C1 expansion (background, parallel with player gameplay)
  └─ after player picks waypoint, main loop fires C2(waypoint_N, encounter=0) (background)

Play encounter 0
  └─ consume_effects(N, 0) → wait for C2 result → inject effects + tolls
  └─ display options → player chooses
  └─ C2 update (bearing + N, next=1) (background)

Play encounter 1 (last)
  └─ consume_effects(N, 1) → wait for C2 result → inject effects + tolls
  └─ player chooses
  └─ C2 update (bearing only) → update entry_id only

Waypoint N ends, player selects waypoint N+1
  └─ C2 update (bearing + N+1, encounter=0) (background)
  └─ enter waypoint N+1, prefetch result already ready
```

## Tendency Language

State passed from C0 → C2 uses "tendency descriptions" (A0 precise values compressed to tendency bands), example:

```
## Current state (tendency)
  health:  high (stable)
  sanity:  moderate (falling)
  depth:   2,  act: 2  active_marks: lamp_oil

## Waypoint trajectory (2 completed)
  [crossroads] depth=1  sanity_delta=-1  flags=none
  [market]     depth=2  sanity_delta=-2  flags=witnessed_violence

## Active waypoint so far (partial)
  encounters_resolved=1  sanity_lost=1

## Effect delta calibration (calibrate h/m/s to current state)
  h (health  high/100): [-18, +5]  — reserve extremes for pivotal options
  m (money   moderate): [-4, +8]
  s (sanity  moderate/100): [-12, +5]  — fragile sanity → smaller losses

Assign effects for: waypoint_id=archive_vault, encounter_index=1
```

C2 returns `{"entry_id": N, "effects": [{"id": "opt_a", "h": -2, "m": 0, "s": -1, "toll": "destabilizing"}, ...]}`.

The `## Effect delta calibration` section is generated per-call by `collector.py` from current tendency bands. It tells C2 the proportional delta range for each resource given the player's actual state — preventing absurd output values. C2 may exceed the suggested range for pivotal options, but large swings should be reserved for critical choices.

## Content Boundaries

### LLM may generate

- Saga graph, waypoint topology, map_blurb, toll lexicon, rules, narration_table (C3, `./loombound gen`)
- scene_concept, sanity_axis, option intents (C2, A1 cache table generation)
- Bearing entry_id + per-option h/m/s values + tolls (C2, runtime)
- scene_summary, sanity_question, option labels, add_events (C1, runtime expansion)

### LLM may not write directly

- CoreState numeric field application (executed by kernel)
- Final RuleTemplate selection
- Structural fields of Run / Waypoint / Encounter

## Offline Generation Flow

```bash
# 1. Generate global bearing enumeration (one-time)
./loombound arc-palette

# 2. Generate saga (Opus builds graph + Haiku builds A1 cache, automatic)
./loombound gen "theme" --lang zh

# 3. Play
./loombound run --lang zh
```
